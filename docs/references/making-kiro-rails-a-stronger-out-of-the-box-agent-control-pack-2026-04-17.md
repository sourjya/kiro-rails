# Making kiro-rails a stronger out-of-the-box agent control pack

## Executive take

kiro-rails is already a serious guardrail template. The repository advertises 14 steering files, 4 automated hooks, 2 review prompts, 1 security agent, 1 TDD task template, and a 13-directory docs taxonomy, with an installer that updates managed files while preserving a single customization file. But most of that strength is concentrated in always-on steering and advisory prompts. The next major upgrade should not be ŌĆ£more static markdown.ŌĆØ It should be a layered control plane: short always-on rules for invariants, on-demand skills and workflows for procedures, powers for MCP-backed integrations, deterministic shell checks for things machines can verify, and cross-tool exports so one canonical source can feed Kiro, Claude Code, Windsurf, Cline, Cursor-style rule systems, and GitHub Copilot. That is where the ecosystem is converging, and it is also where the research points: persistent context is useful, but bloated or auto-generated context files can raise cost and even hurt success rates. ## Where the current .kiro implementation is leaving power on the table

The biggest structural gap is simple: there is no real capability layer. The `.kiro/` tree ships `steering/`, `hooks/`, `agents/`, `prompts/`, `specs/`, and `templates/`, but no `skills/` and no powers. Kiro explicitly distinguishes steering, skills, and powers, and its own docs say powers are usually the better fit for MCP-backed integrations because they bundle tools with guidance and activate automatically by context. Right now kiro-rails tells the agent how to behave, but it gives the agent very few packaged procedures, scripts, or tool bundles to execute. That is the most obvious place where the pack is behind the destination platform itself. ŅłĆciteŅłéturn5view0Ņłéturn30view0Ņłéturn30view2Ņłéturn31view0Ņłü

The hook suite is useful but still lighter than the opportunity. `lint-python-files.kiro.hook` runs `uvx ruff check --fix .` against the whole repo on any Python file edit. That is convenient for tiny repos, but in larger codebases it is expensive and can generate unrelated changes. `security-checkpoint.kiro.hook` only watches a narrow set of paths and includes a Pydantic-specific rule, which makes the pack feel Python-centric. `comment-standards-check.kiro.hook` is wired through `preToolUse` on shell commands that look like `git commit`, which means it is really policing shell-driven commit flows, not commits as a first-class repository event. `changelog-maintenance.kiro.hook` depends on the agent to author prose instead of using a deterministic fragment or release-note mechanism. Meanwhile, Kiro hooks can fire on a much richer set of events, including agent turn completion and before or after spec task execution, and none of those are being used yet. ŅłĆciteŅłéturn10view0Ņłéturn10view1Ņłéturn10view2Ņłéturn10view3Ņłéturn29view0Ņłü

The prompt and agent story is half-built. The maintainability review prompt is strong and specific, but it is monolithic and prose-heavy. The security review prompt is concise, but almost too compressed for a pack that claims cross-stack utility. You also already have a maintainability review surface as a prompt, but only the security reviewer got promoted to a dedicated agent. The security agent is read-only in practice, which is good, because `shell` appears in its declared tools but not in `allowedTools`. Still, its `resources` list is narrow for a general-purpose template: it includes Python and Node package files plus Docker artifacts, but omits things like `Gemfile`, CI workflows, Terraform, Kubernetes, Helm, and other common security-relevant files. ŅłĆciteŅłéturn7view0Ņłéturn7view1Ņłéturn7view2Ņłéturn9view1Ņłéturn9view2Ņłü

Some of the always-on steering is probably too absolute for broad adoption. `code-commenting-standards.md` requires documentation for every class, property, and function at all visibility levels. `reusable-architecture.md` pushes an absolute ŌĆ£ZERO embedded literalsŌĆØ line. Those instincts are understandable, but recent ETH Zurich work found that context files work best when they carry non-inferable, human-authored details, not when they become a second source tree full of boilerplate or globally enforced verbosity. In the current form, kiro-rails risks making agents spend too much attention on documentation theater in places where better naming and tests would do more work. ## What the major agentic tools are converging on

The biggest convergence pattern is dynamic loading over static monoliths. Kiro skills load only name and description at startup, then load full instructions only when relevant; Kiro powers do the same while also bundling MCP tools and workflow guidance. Claude Code skills follow the same progressive-disclosure idea, and Claude adds richer controls like `allowed-tools`, arguments, and `context: fork` for isolated subagent execution. The common design is obvious: do not stuff every playbook into the system prompt. Keep startup context cheap, then load the procedure, references, or scripts only when the task actually needs them. ŅłĆciteŅłéturn21view0Ņłéturn21view1Ņłéturn21view2Ņłéturn30view0Ņłéturn30view2Ņłéturn32view0Ņłéturn32view1Ņłéturn32view3Ņłéturn32view4Ņłéturn32view5Ņłü

Cursor, Windsurf, and Cline are converging on the same architecture even though the names differ. Cursor has rules, skills, plan mode, dynamic context discovery, cloud agents, and sandboxing. Windsurf splits durable context across rules and `AGENTS.md`, keeps workflows manual via slash commands, treats skills as dynamic procedures with supporting files, and warns users not to rely on auto-generated memories for durable shared knowledge. Cline similarly separates persistent rules from Plan/Act execution and a structured Memory Bank methodology, and it already supports multiple rule formats from other ecosystems, including `.cursorrules`, `.windsurfrules`, and `AGENTS.md`. If you zoom out, all of these products are building the same four-layer stack: invariant rules, reusable procedures, transient plans, and tool-backed execution. ŅłĆciteŅłéturn24view3Ņłéturn25view0Ņłéturn25view1Ņłéturn25view2Ņłéturn26view0Ņłéturn26view1Ņłéturn26view2Ņłéturn26view3Ņłéturn27view1Ņłéturn27view2Ņłéturn27view3Ņłü

GitHub Copilot is reinforcing the portability side of that trend. It supports repository-wide instructions, path-specific instruction files, `AGENTS.md` files for agents, and custom agent profiles that define description, prompts, tools, model, and MCP configurations. That matters because it means a future-proof pack should not think in terms of ŌĆ£what Kiro can read,ŌĆØ but ŌĆ£what this project should expose to any agent surface.ŌĆØ A canonical manifest that compiles into Kiro steering, Copilot instructions, AGENTS.md, and sibling file formats is now a better product direction than manually maintaining separate trees. ŅłĆciteŅłéturn28view0Ņłéturn28view1Ņłéturn28view2Ņłéturn28view3Ņłü

These products are also getting popular for concrete reasons, not vibes and smoke machines. Kiro frames powers as a fix for MCP context overload. Cursor says sandboxed agents stop 40% less often than unsandboxed ones, which directly addresses approval fatigue. Anthropic has written that Claude is increasingly handling more complex engineering work more autonomously, with internal usage shifting toward design and planning as well as implementation, and it reported the product generally handles around 20 actions before human input where it used to handle around 10. In other words, the more autonomy tools deliver, the more valuable project-level controls become. That is why these control surfaces are gaining traction. ## The highest-leverage upgrades for this pack

The top recommendation is to build a single-source control compiler. Keep one canonical project manifest, probably a small structured file plus human-authored override sections, and generate `AGENTS.md`, Kiro steering fragments, CLAUDE.md or skill scaffolds, Copilot repo instructions, Copilot path-specific instruction files, Windsurf rules, and Cline rules from it. That move lines up with the spread of AGENTS.md, with CopilotŌĆÖs multi-surface instruction system, and with KiroŌĆÖs support for AGENTS.md alongside native steering. It also solves the exact real-world confusion showing up on Stack Overflow. ŅłĆciteŅłéturn34search2Ņłéturn34search14Ņłéturn28view0Ņłéturn28view3Ņłéturn29view1Ņłéturn38view0Ņłü

The second recommendation is to add an actual capability layer. Right now kiro-rails is a rules pack. It needs to become a rules-plus-procedures pack. I would ship a small set of portable skills or workflows immediately: `feature-spec-author`, `bug-triage`, `playwright-regression`, `api-contract-review`, `incident-debug`, `release-manager`, `dependency-upgrade-audit`, and `migration-planner`. Use powers only where MCP-backed tools are genuinely part of the solution, like GitHub, Postman, Figma, Stripe, Datadog, Terraform, or cloud platforms. Kiro explicitly recommends powers for MCP-heavy integrations, and Windsurf makes a similar distinction between skills and workflows. ŅłĆciteŅłéturn30view0Ņłéturn30view2Ņłéturn31view0Ņłéturn31view3Ņłéturn26view0Ņłéturn26view1Ņłéturn26view2Ņłü

The third recommendation is to make the verification lane deterministic. Keep advisory hooks, but give them machine teeth. Secret scanning, dependency and license audit, changed-file lint and typecheck, nearest-test execution, API schema diffing, broken-link and ADR-link validation, docs drift detection, and release-note fragment collation should be scripts first and prompts second. Kiro hooks can already run shell commands and can fire on more events than you currently use. AnthropicŌĆÖs guidance on tool writing also pushes toward deterministic, evaluated tools over wishful prompting. ŅłĆciteŅłéturn29view0Ņłéturn10view0Ņłéturn10view2Ņłéturn33search7Ņłéturn33search12Ņłü

The fourth recommendation is to promote prompts into a real agent roster. You already have a security agent and a maintainability prompt. I would add at least four more agent profiles: `code-maintainability-reviewer`, `architecture-reviewer`, `test-failure-triager`, and `release-readiness-reviewer`. Give each one narrower resources, a structured output schema, and explicit allowed tools. Keep the security reviewer read-only by default, but expand its resource globs to cover stack-relevant files and CI or infra config. The point is not to multiply personas for fun. It is to make common review passes reusable, bounded, and consistent. ŅłĆciteŅłéturn7view0Ņłéturn7view1Ņłéturn7view2Ņłéturn9view1Ņłéturn9view2Ņłü

The fifth recommendation is to add a lightweight memory and plan layer, but keep it short and human-written. CursorŌĆÖs saved plans and ClineŌĆÖs Memory Bank both solve a real problem: agents lose task-local understanding across sessions and compaction. At the same time, the ETH paper is a warning not to turn memory files into bloated pseudo-architecture books. So the right move is a minimal `memory-bank/` or `docs/context/` with current focus, known quirks, critical commands, and current progress. Think ŌĆ£handoff file,ŌĆØ not ŌĆ£encyclopedia.ŌĆØ ŅłĆciteŅłéturn25view1Ņłéturn27view2Ņłéturn34search1Ņłéturn34search4Ņłü

The sixth recommendation is stack presets. The installer already asks for stack and database details, but the shipped controls are still mostly Python-shaped. Split the pack into preset modules for Python/FastAPI, Ruby/Rails, Node/Next, TypeScript library, Go service, and monorepo shapes. Each preset should own its hooks, security checks, test layout defaults, dependency tool guidance, and resource globs. That will make the pack more credible and more immediately useful to real teams because the out-of-the-box behavior will actually match their runtime.
