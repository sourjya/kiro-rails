# Requirements: Bug Scribe

## Functional Requirements

### REQ-1: Marker Detection

**Description:** The system detects `# bug:` or `// bug:` markers in source files when they are saved.

**Acceptance Criteria:**
- [ ] Detects markers in Python files (`# bug: ...`)
- [ ] Detects markers in JS/TS/Go/Rust/Java files (`// bug: ...`)
- [ ] Detects markers in CSS/SCSS files (`/* bug: ... */`)
- [ ] Case-insensitive on trigger word (`BUG`, `bug`, `Bug` all fire)
- [ ] Case-insensitive on category (normalized to UPPER_SNAKE_CASE in output)
- [ ] Requires space after comment prefix, space after colon, em-dash separator
- [ ] Does NOT fire on malformed markers (missing space, missing em-dash)
- [ ] Produces a helpful warning on near-miss markers (detected but structurally invalid)

### REQ-2: Bug Doc Scaffolding

**Description:** When a valid marker is detected, the system creates a `docs/bugs/BUG-###-category.md` file from the existing template with pre-filled fields.

**Acceptance Criteria:**
- [ ] Determines the next sequential BUG number from existing files in `docs/bugs/`
- [ ] Creates the file with correct naming: `BUG-###-category.md` (category lowercased in filename)
- [ ] Pre-fills: ID, Category, File path, Date, Branch name, Status (IN_PROGRESS)
- [ ] Injects the current `git diff` for the triggering file into the doc
- [ ] Injects surrounding code context (±5 lines around the marker)
- [ ] Leaves human-judgment fields (Root Cause, Fix Description, Severity) as placeholders
- [ ] Uses `sed` substitution on the existing `BUG-000-template.md` — does not hardcode the template structure

### REQ-3: Chokepoint Log Entry

**Description:** When a valid marker is detected, the system appends an entry to `docs/engineering/chokepoint-log.md`.

**Acceptance Criteria:**
- [ ] Appends a new entry with the BUG ID, category, file, date, and description
- [ ] Uses the existing chokepoint log format (`### CP-NNN: ...`)
- [ ] Determines the next CP number from existing entries
- [ ] Marks the entry as "auto-generated by Bug Scribe" so manual entries are distinguishable

### REQ-4: Category Validation

**Description:** The system validates the marker's category against the known taxonomy and warns on unknown categories.

**Acceptance Criteria:**
- [ ] Known categories fire without warning
- [ ] Unknown categories still fire (don't block) but log a warning message
- [ ] Warning message suggests adding the category to the taxonomy if it recurs
- [ ] Known category list is read from a config source (not hardcoded in the script)

### REQ-5: Near-Miss Detection

**Description:** Common mis-typed markers are detected and produce a helpful message without triggering automation.

**Acceptance Criteria:**
- [ ] `#BUG:` (no space after hash) produces a warning with the correct format
- [ ] `# BUG:stuff` (no space after colon) produces a warning
- [ ] `# BUG: category - description` (hyphen instead of em-dash) produces a warning
- [ ] Warning messages show the expected format
- [ ] Near-misses do NOT create bug docs or chokepoint entries

### REQ-6: Pattern Detection (Agent-Powered, Optional)

**Description:** When a bug doc is created or updated, an optional agent hook analyzes the chokepoint log for recurring patterns.

**Acceptance Criteria:**
- [ ] Fires when any `docs/bugs/BUG-*.md` file is edited
- [ ] Agent reads the chokepoint log and counts entries per category
- [ ] When 2+ bugs share a category, the agent flags the trend in its response
- [ ] When 3+ bugs share a category, the agent recommends guardrail promotion (per existing chokepoint-logging.md rules)
- [ ] Hook can be disabled without affecting Hook 1

### REQ-7: Idempotency

**Description:** Re-saving a file with an existing marker does not create duplicate bug docs.

**Acceptance Criteria:**
- [ ] If the marker text already has a corresponding bug doc (matched by file + marker content), the hook is a no-op
- [ ] The detection mechanism tracks which markers have been processed (options: marker line checksum log, or checking if a bug doc already references this file+category combination)

## Non-Functional Requirements

- **Performance:** Hook 1 completes in <1 second (shell script, no network calls)
- **Zero dependencies:** Hook 1 requires only bash, sed, awk, git — no Python, no npm, no external tools
- **Cross-platform:** Works on Linux and macOS (bash 4+). Windows users need Git Bash/WSL.
- **Observability:** Every action logs what it did to stdout (picked up by Kiro hook output)
- **Non-destructive:** Never overwrites an existing bug doc. If a number collision occurs, increment until a free slot is found.
